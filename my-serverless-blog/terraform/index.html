🚀 Dağıtım (Deployment) Detayları

Bu altyapı, **Terraform** ile AWS **us-east-1** bölgesinde kurulmuştur.

  • **Serverless API:** AWS Lambda (Node.js) + API Gateway (REST API)
  • **Dokümantasyon/Blog:** AWS S3 Statik Web Sitesi
  • **IAM:** En az ayrıcalık (Least Privilege) prensibine göre ayarlanmıştır.

💬 Ödev Sorularının Cevapları

1. Neden serverless tercih ettin?

Serverless mimariyi tercih etmemin temel nedenleri **maliyet etkinliği** ve **operasyonel yükün düşürülmesidir**. Sunucu yönetimi (yama, işletim sistemi bakımı) ortadan kalktığı için tamamen koda odaklanabildim. Projenin trafik hacmi düşük olduğu için, sadece fonksiyon çalıştığında ödeme yapmak maliyeti sıfıra yaklaştırmaktadır. Ayrıca, otomatik ölçeklenme yeteneği, gelecekteki trafik artışları için endişe etmemi engeller.

2. Karşılaştığın en büyük zorluk neydi?

En büyük zorluk, **Terraform state yönetimi ve AWS IAM/API Gateway entegrasyon hataları** oldu. Özellikle:

  1. İlk başta aldığım **"Not Found"** hataları (API Gateway'in Lambda'yı çağırma izninin [Lambda Permission] eksikliğinden kaynaklanıyordu).
  2. Farklı API Gateway tipleri (REST vs. HTTP API) arasındaki state çakışmaları.
  3. S3'te statik site için dosya yüklerken karşılaştığım **ACL (Access Control List)** hatası (Modern AWS S3 ayarlarıyla uyumsuzluk).
Bu zorluklar, Least Privilege (En Az Ayrıcalık) prensibini uygulamayı öğrenmemi sağladı.

3. Bu yaklaşım hangi gerçek dünya senaryolarında işe yarar?

Serverless mimari, özellikle **olay tabanlı (event-driven)** ve **istek tabanlı (on-demand)** iş yükleri için mükemmeldir:

  • **Hafif API/Mikroservisler:** Geleneksel sunucu gerektirmeyen, basit veri okuma/yazma işlemleri (Bu ödevdeki gibi).
  • **Asenkron Veri İşleme:** Bir kullanıcı S3'e resim yüklediğinde, Lambda'nın otomatik tetiklenerek resmi yeniden boyutlandırması veya meta verilerini işlemesi.
  • **Cron İşleri:** Veritabanı temizliği, günlük rapor oluşturma gibi düzenli, zamanlanmış görevler.
